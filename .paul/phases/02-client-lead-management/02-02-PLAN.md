## Goal
Build the client detail page at `/dashboard/clients/[id]` — showing a full client profile with inline edit and delete-with-confirmation, and making the client list rows clickable.

## Purpose
Completes Phase 2: the AYS team can now not only create and list clients, but open any client, correct a phone number, update a source tag, or delete a test entry. Every interaction is fast and in-place — no separate edit page.

## Output
- `/dashboard/clients/[id]` detail page (Server Component, dynamic route)
- Inline edit mode: "Edit" button converts display fields to form inputs in place
- Inline delete: "Delete" button shows confirm step before removing client
- Server actions: `getClient`, `updateClient`, `deleteClient`
- Client list rows become clickable links to the detail page
- `loading.tsx`, `error.tsx`, `not-found.tsx` for the detail route
</objective>

  <name>Task 1: Create server actions for client detail, update, and delete</name>
  <files>src/app/(dashboard)/dashboard/clients/[id]/actions.ts</files>
  <action>
    Create `src/app/(dashboard)/dashboard/clients/[id]/actions.ts` with three server actions:

    **`getClient(id: string): Promise<Client | null>`**
    - Query `clients` table by `id`
    - Return the row, or `null` if not found (let page.tsx call `notFound()`)
    - On DB error: throw (error.tsx catches it)

    **`updateClient(id: string, formData: FormData): Promise<{ success: true } | { error: string }>`**
    - Extract and validate identically to `addClient` in 02-01:
      - name: trim, non-empty required
      - phone: trim, nullable
      - email: trim, lowercase, nullable; validate format if provided
      - source: normalize to ClientSource, default 'direct'
    - Update via Supabase: `.from('clients').update({...}).eq('id', id).select('id')`
      (`.select('id')` causes Supabase to return the affected rows so we can detect 0-row updates)
    - If `data` is an empty array: return `{ error: 'Client not found. It may have been deleted.' }`
      (silent no-op guard — prevents stale session from claiming success on a deleted record)
    - Catch code `23505` → return `{ error: 'A client with this email already exists.' }`
    - On other DB errors: log server-side (no PII), return generic message
    - On success: `revalidatePath('/dashboard/clients')`, `revalidatePath(\`/dashboard/clients/${id}\`)`, return `{ success: true }`

    **`deleteClient(id: string): Promise<void>`**
    - Delete via Supabase: `.from('clients').delete().eq('id', id)`
    - On DB error: throw (error.tsx handles surface)
    - On success: call `revalidatePath('/dashboard/clients')`, `revalidatePath('/dashboard')`
      (dashboard stat card count must reflect the deletion), then `redirect('/dashboard/clients')`
    - Import `redirect` from `'next/navigation'`

    All three functions: `'use server'` directive at top of file.
    Avoid: Any type. Avoid: Logging PII.
  </action>

  <action>
    **`src/app/(dashboard)/dashboard/clients/[id]/page.tsx`** — Server Component:
    - Props: `{ params: Promise<{ id: string }> }` (Next.js 16: params is async)
    - Call `const { id } = await params`
    - Call `getClient(id)` — if null, call `notFound()` from `'next/navigation'`
    - Render `<ClientDetail client={client} />`

    **`src/components/clients/ClientDetail.tsx`** — Client Component:
    Props: `{ client: Client }`

    Two modes controlled by `useState<'view' | 'edit' | 'confirm-delete'>('view')`:

    **VIEW mode:**
    - Back link: `← Clients` linking to `/dashboard/clients`
    - Page heading: client name (large, bold)
    - Source badge (reuse same SOURCE_CLASSES/SOURCE_LABELS pattern from ClientList)
    - Detail rows: Phone, Email, Source, Date Added — each in a labeled row
    - Null values display as `—`
    - Action row (top-right of card): `[Edit]` button + `[Delete]` button (red outline)

    **EDIT mode:**
    - Same layout but fields become form inputs (pre-filled with current values)
    - Fields: Name (required), Phone (tel, placeholder "+1 868 123 4567"), Email, Source (select)
    - Inline errors for name and email (same validation pattern as CreateClientModal)
    - Submit calls `updateClient(client.id, formData)` via `useTransition`
    - On `{ error }`: show error, stay in edit mode
    - On `{ success }`: call `router.refresh()` (re-fetches the Server Component with updated DB data,
      so view mode shows new values immediately), then `setMode('view')`
      Import `useRouter` from `'next/navigation'`; call `const router = useRouter()` at component top.
      `router.refresh()` is NOT a full page reload — it re-runs server data fetching without navigation.
    - Buttons: `[Save]` (disabled while pending, shows "Saving...") + `[Cancel]` (returns to view mode, discards changes)
    - Avoid: `any` types. Avoid: full page reload on success (`router.refresh()` is sufficient).

    **CONFIRM-DELETE mode** (triggered by clicking Delete in view mode):
    - Show inline warning box: "Are you sure? Deleting [client name] cannot be undone."
    - Buttons: `[Confirm Delete]` (red, calls `deleteClient(client.id)`, uses `useTransition`) + `[Cancel]` (returns to view mode)
    - While deleting: disable both buttons, show "Deleting..."
    - `deleteClient` server action handles the redirect server-side
