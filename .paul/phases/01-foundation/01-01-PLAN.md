<objective>
## Goal
Scaffold the AYS CRM as a Next.js 14 App Router application with Supabase Auth, producing a working login page that redirects authenticated users to a protected dashboard shell.

## Purpose
This is the entry point for all future phases. Every feature (clients, pipeline, notes, integrations) builds on this foundation. Getting auth and project structure right here prevents rewrites later.

## Output
- Fully configured Next.js 14 + TypeScript project with security headers
- .gitignore with secrets excluded
- Supabase client helpers (browser + server + middleware)
- Login page (email/password via Supabase Auth) with server-side error logging
- Protected `/dashboard` route (redirects to login if unauthenticated)
- `.env.local.example` documenting all required env vars
</objective>

<context>
## Project Context
@.paul/PROJECT.md
@.paul/STATE.md

## Stack Decisions
- Next.js 14 App Router (not Pages Router)
- Supabase Auth with SSR package (@supabase/ssr) — NOT the legacy @supabase/auth-helpers-nextjs
- TypeScript strict mode
- Deployed to Vercel (serverless — no persistent Node processes)
- Tailwind CSS for styling
</context>

    Initialize a Next.js 14 App Router project with TypeScript and Tailwind CSS.

    - @supabase/supabase-js
    - @supabase/ssr  ← use this, NOT @supabase/auth-helpers-nextjs (deprecated)

    **Create .gitignore** (CRITICAL — must exist before any git init):
    ```

    node_modules/

    # Environment variables — NEVER commit these
    .env
    .env.local
    .env.*.local

    # Next.js build output
    .next/
    out/

    # Vercel
    .vercel

    # Misc
    .DS_Store
    *.pem
    ```

    **Create .env.local.example** (placeholder values only — safe to commit):
    ```
    NEXT_PUBLIC_SUPABASE_URL=https://your-project-ref.supabase.co
    NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key-here
    ```

    **Configure next.config.ts with security headers:**
    ```ts
    import type { NextConfig } from 'next'

    const securityHeaders = [
      { key: 'X-Frame-Options', value: 'SAMEORIGIN' },
      { key: 'X-Content-Type-Options', value: 'nosniff' },
      { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
      { key: 'Permissions-Policy', value: 'camera=(), microphone=(), geolocation=()' },
    ]

    const nextConfig: NextConfig = {
      async headers() {
        return [
          {
            source: '/(.*)',
            headers: securityHeaders,
          },
        ]
      },
    }

    export default nextConfig
    ```

    **Supabase Auth configuration note:**
    In the Supabase dashboard → Authentication → Settings:
    - Email confirmation: DISABLE for this project (CRM users are manually created by admin, not self-registered)
    - This must be explicitly set; Supabase enables email confirmation by default

    Avoid: App Router with `output: 'export'` (breaks server components + middleware)
  </action>

<task type="auto">
  <name>Task 2: Create Supabase client helpers</name>
  <files>src/lib/supabase/client.ts, src/lib/supabase/server.ts, src/lib/supabase/middleware.ts</files>
  <action>
    Create three Supabase client helpers using @supabase/ssr:

    **src/lib/supabase/client.ts** — browser client (for Client Components):
    ```ts
    import { createBrowserClient } from '@supabase/ssr'
    export function createClient() {
      return createBrowserClient(
        process.env.NEXT_PUBLIC_SUPABASE_URL!,
        process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
      )
    }
    ```

    **src/lib/supabase/server.ts** — server client (for Server Components + Route Handlers):
    Use `createServerClient` from @supabase/ssr with Next.js `cookies()` from `next/headers`.
    Must be async (Next.js 15 cookies() is async).

    **src/lib/supabase/middleware.ts** — middleware client:
    Use `createServerClient` with request/response cookie handlers for session refresh in middleware.

    Note on cookie security: @supabase/ssr sets HttpOnly and SameSite=Lax on auth cookies by default.
    On Vercel (HTTPS), the Secure flag is set automatically by the platform. No additional cookie config needed.

    Avoid: Creating a single global client instance — each request needs its own client for SSR cookie isolation.
  </action>
  <verify>
    TypeScript compiles: `npx tsc --noEmit` passes.
    No imports from @supabase/auth-helpers-nextjs anywhere.
    `grep -r "auth-helpers-nextjs" src/` returns no results.
  </verify>
  <done>AC-1 satisfied: Supabase clients configured correctly for SSR</done>

<task type="auto">
  <name>Task 3: Create middleware for route protection</name>
  <files>src/middleware.ts</files>
  <action>
    Create Next.js middleware that:
    1. Refreshes Supabase session on every request (required by @supabase/ssr)
    2. Redirects unauthenticated users from /dashboard/* to /login
    3. Redirects authenticated users away from /login to /dashboard

    **Matcher config** — must explicitly exclude API routes to avoid blocking future webhooks:
    ```ts
    export const config = {
      matcher: [
        /*
         * Match all request paths EXCEPT:
         * - _next/static (static files)
         * - _next/image (image optimization)
         * - favicon.ico
         * - /api/* (webhook and API routes must not be redirected)
         * - public assets
         */
        '/((?!_next/static|_next/image|favicon.ico|api/|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)',
      ],
    }
    ```

    Use the middleware Supabase client from src/lib/supabase/middleware.ts.

    Avoid: Blocking /api/* routes — future Gmail and WhatsApp webhook endpoints must bypass auth middleware.
  </action>
  <verify>
    Navigate to /dashboard without a session → redirects to /login.
    After login → redirects to /dashboard.
    `curl -I http://localhost:3000/api/health` (even if 404) should NOT return a redirect to /login — confirms /api/* is excluded.
  </verify>

  <name>Task 4: Create login page with email/password auth and error logging</name>
  <files>src/app/(auth)/login/page.tsx, src/app/(auth)/login/actions.ts, src/app/page.tsx, src/app/layout.tsx</files>

    **src/app/page.tsx** — root redirect:
    Redirect to /dashboard (middleware handles auth check).

    **src/app/layout.tsx** — root layout:
    Add html/body, import Tailwind globals.

    **src/app/(auth)/login/page.tsx** — login form:
    - Email + password fields
    - Submit button (disabled during submission to prevent double-submit)
    - Error message display area
    - Calls server action from actions.ts
    - Clean, simple layout (no need for complex styling yet)

    **src/app/(auth)/login/actions.ts** — server actions:
    - `login(formData)`:
      1. Validate that email and password fields are non-empty before calling Supabase (avoid unnecessary API calls)
      2. Call supabase.auth.signInWithPassword
      3. On success: redirect to /dashboard
      4. On auth failure (invalid credentials): return user-facing error message "Invalid email or password"
      5. On unexpected error (network, service unavailable): log to console.error with timestamp + sanitized context (never log passwords), return generic message "Something went wrong, please try again"
    - `logout()`: call supabase.auth.signOut, redirect to /login

    **Error logging format for unexpected errors:**
    ```ts
    console.error(`[AUTH] Login error at ${new Date().toISOString()}:`, {
      errorCode: error.code,
      errorMessage: error.message,
      // Never log email or password
    })
    ```

    Avoid: Client-side auth state management — use server actions + redirects for simplicity.
    Avoid: Exposing raw Supabase error messages to the user — map to user-safe messages.
    Avoid: Logging PII (email addresses) in error output.
  </action>
  <verify>
    Submit valid credentials → redirected to /dashboard.
    Submit invalid credentials → error message "Invalid email or password" shown, stays on /login.
    Simulate service error → generic message shown, console.error entry visible in server logs.
    Empty form submission → client-side validation prevents API call.
  </verify>
  <done>AC-3 and AC-4 satisfied: login works with valid/invalid credentials with proper error handling and logging.</done>
